# 🧑‍💼 Subagent Virtual Employee System

A runnable MVP for a **digital virtual-employee system**. You are the *manager*
(the main agent). You create and manage a team of AI *employees* (subagents),
give each one a persona and a personal knowledge base, pull them into meetings,
and assign them collaborative goals — then read the transcripts, minutes, and
reports they produce.

It runs **fully offline with zero API keys**: every "subagent" contribution is
generated by a deterministic, persona-driven engine. An optional live-LLM path
(Anthropic) can be switched on with an env var, but nothing requires it.

---

## ✨ What it does (the six MVP flows)

| # | Capability | Where |
|---|------------|-------|
| 1 | **Create an employee role** — name, role, personality, expertise, objectives, comms style, plus an auto-generated background you can edit | Employees → *New employee* |
| 2 | **Personal knowledge base** — add/remove notes & snippets per employee; used when they reason in meetings and goals | Employees → open a card → *knowledge base* |
| 3 | **Meeting orchestration** — pick employees, set a topic and round count, run a simulated discussion → **transcript + minutes + report** | Meetings |
| 4 | **Goal assignment** — assign a goal to one or more employees; work is split by expertise into tasks with status + a **collaboration output** | Goals |
| 5 | **New-role ideation** — describe the employee you want; the system drafts a full profile you can edit before saving | Employees → *Ideate a role* |
| 6 | **Persistence** — everything is saved to a JSON file and survives restarts | `server/data/db.json` |

## 🚀 Quick start

Requires **Node.js ≥ 18** (built and tested on Node 22).

```bash
npm install          # install deps
npm run seed         # load sample employees + a demo meeting (optional)
npm run dev          # start API (:3001) + web (:5173) with hot reload
```

Then open **http://localhost:5173**.

### Production-style run (single server)

```bash
npm run serve        # builds the client, then serves API + UI on :3001
# open http://localhost:3001
```

### Other scripts

```bash
npm test             # end-to-end smoke test (boots the app, exercises every flow)
npm run build        # build the client into client/dist
npm start            # serve an already-built client + API on :3001
```

## 🏛️ Architecture

```
subagent-virtual-employee-system/
├── server/                     Express API (Node, ESM)
│   ├── src/
│   │   ├── index.js            HTTP routes; serves client/dist in production
│   │   ├── engine.js  ★        the "subagent" reasoning — pure, offline generators
│   │   ├── llm.js              optional Anthropic path (fallback to engine)
│   │   ├── db.js               tiny JSON-file store with atomic writes
│   │   └── seed.js             sample data
│   ├── data/db.json            persisted state (git-ignored)
│   └── test/smoke.mjs          end-to-end HTTP smoke test
└── client/                     Vite + React SPA
    └── src/
        ├── App.jsx             shell + tab navigation
        ├── api.js              fetch wrapper
        ├── components/ui.jsx   modal, markdown renderer, employee picker
        └── pages/              EmployeesPage, MeetingsPage, GoalsPage
```

**Stack:** Vite + React (client) · Express (API) · a hand-rolled JSON-file store
(no native modules, nothing to compile). Chosen for the fastest path to a
runnable local MVP.

### How the "subagents" think

`server/src/engine.js` is the heart of the product. Each employee is a persona
(role, expertise, personality, communication style) plus a personal knowledge
base. The engine turns those into behavior:

- **`generateProfile`** — composes a narrative background from the form fields.
- **`ideateRole`** — matches a free-text description to a role archetype and
  drafts a complete, editable profile.
- **`runMeeting`** — each participant "speaks" over N rounds. Contributions are
  flavored by their role and expertise, reference their knowledge notes, and
  react to prior speakers. Produces a structured **transcript**, **minutes**
  (agenda, key points, decisions, action items), and a markdown **report**.
- **`executeGoal`** — decomposes a goal across assignees by expertise into
  tasks, then writes an integrated **collaboration output**.

Because the engine is pure and deterministic, the app is instant, free, and
works with no network — ideal for an MVP demo. To upgrade any of these to real
LLM output, set an API key (below); the routes call Claude and gracefully fall
back to the engine on any error.

### Optional: live LLM

```bash
export ANTHROPIC_API_KEY=sk-ant-...        # enables richer generation
export ANTHROPIC_MODEL=claude-sonnet-5     # optional, this is the default
npm run dev
```

The header pill flips from **"LLM: simulated"** to **"LLM: live"**. Meetings,
goals, and ideation then use Claude; if a request fails, they silently fall back
to the deterministic engine so the app never breaks.

## 🔌 API reference

| Method | Endpoint | Purpose |
|--------|----------|---------|
| GET | `/api/health` | status + counts + LLM mode |
| GET/POST | `/api/employees` | list / create employees |
| GET/PUT/DELETE | `/api/employees/:id` | read / update / delete |
| POST | `/api/employees/generate-profile` | draft a background from fields |
| POST | `/api/employees/ideate` | draft a full role from a description |
| GET/POST | `/api/employees/:id/knowledge` | list / add knowledge notes |
| DELETE | `/api/knowledge/:id` | delete a note |
| GET/POST | `/api/meetings` | list / run a meeting |
| GET/DELETE | `/api/meetings/:id` | read / delete a meeting |
| GET/POST | `/api/goals` | list / assign a goal |
| PUT/DELETE | `/api/goals/:id` | update status / delete |

## ✅ Validation

`npm test` boots the real Express app on an ephemeral port (against a throwaway
data file so it never touches your saved data) and asserts every core flow:
employee creation + profile generation, required-field validation, role
ideation, knowledge notes, running a meeting (transcript/minutes/report), goal
assignment (task split + output), and persistence.

---

## Notes & limitations

- Single-user, local-first; the JSON store is fine for an MVP but not for
  concurrent multi-user use.
- Meeting "discussion" is generated, not a live multi-agent loop — swap in the
  LLM path for genuine reasoning.
- Data lives in `server/data/db.json`; delete it (or re-run `npm run seed`) to
  reset.
